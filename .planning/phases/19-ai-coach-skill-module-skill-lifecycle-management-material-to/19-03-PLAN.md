1. Create `backend/app/services/skill_validation_service.py`:

   Import: `json, logging, re, dataclasses`

   **@dataclass ValidationIssue:**
   - `severity: str` — "error" | "warning" | "info"
   - `dimension: str` — Check dimension name
   - `message: str` — Human-readable issue description
   - `suggestion: str` — How to fix

   **@dataclass StructureCheckResult:**
   - `passed: bool`
   - `score: int` — 0-100
   - `issues: list[ValidationIssue]`

   **async def check_skill_structure(skill: Skill) -> StructureCheckResult:**

   Pure rule-based checks (no AI call — instant, per D-10):

   a) **SOP Content Check:**
   - Parse skill.content (Markdown body). Look for heading patterns (`## Step` or `### Step`) to count SOP steps.
   - Use regex: `r'^#{2,3}\s+(?:Step|步骤)\s*\d'` on content lines to count steps
   - If steps < 3: error "SOP 步骤少于 3 步，培训流程不完整" suggestion "补充缺失的对话阶段（开场、需求挖掘、产品介绍、异议处理、收尾）"
   - Check for key stages by keyword matching in content:
     - REQUIRED_STAGES = [("Opening", "开场"), ("Product", "产品介绍", "产品"), ("Closing", "收尾", "总结")]
     - For each stage: if no keyword found in content (case-insensitive): warning "{stage} 相关阶段可能缺失"

   b) **Assessment Criteria Check:**
   - Look for "Assessment" or "考核" or "Rubric" sections in content
   - Count assessment items (lines starting with `|` in table rows, or `-` items under assessment heading)
   - If items < 2: error "考核项少于 2 项，无法多维度评估"
   - Try to parse weight numbers; if found and don't sum to ~100: error "考核权重总和应为 100"

   c) **Knowledge Points Check:**
   - Look for "Knowledge" or "知识" section heading in content
   - Count knowledge items (sub-headings or list items under that section)
   - If items < 1: warning "无关键知识点"

   d) **Basic Info Check:**
   - Check skill.name: if empty or len < 2: error "Skill 名称为空或过短"
   - Check skill.description: if empty or len < 10: warning "Skill 描述过短"
   - Check skill.content: if empty or len < 50: error "Skill 内容为空或过短"

   e) **Score Calculation:**
   - Start at 100
   - Each "error" severity: -25 points
   - Each "warning" severity: -10 points
   - Minimum 0
   - passed = (error_count == 0)

   Return StructureCheckResult(passed=passed, score=score, issues=issues)

   **def to_dict(result: StructureCheckResult) -> dict:**
   - Convert to serializable dict: {"passed": bool, "score": int, "issues": [{"severity": ..., "dimension": ..., "message": ..., "suggestion": ...}]}

     settings = get_settings()

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Skill content -> LLM | Untrusted user content sent to Azure OpenAI for evaluation |
| LLM response -> DB | AI-generated scores stored as authoritative quality metrics |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-09 | Tampering | SOP content injection via crafted documents | mitigate | L1 structure check validates content structure before L2 AI evaluation. L2 prompt instructs AI to evaluate objectively, not execute content. |
| T-19-10 | Elevation | Quality gate bypass | mitigate | publish_skill() enforces L1+L2 gate server-side. Frontend warning for 50-69 but actual enforcement is backend. Admin cannot publish without passing backend check. |
| T-19-11 | Denial of Service | L2 evaluation cost | accept | L2 is admin-triggered only, not automatic. Rate limiting via single background task pattern (cannot start new eval while one runs). |
</threat_model>
