# Plan 18-01: Backend File Download API & Storage URL Security Fix

## Goal
Add a secure file download endpoint and fix `storage_url` information disclosure.

## Context
- Files stored at `./storage/materials/{material_id}/v{version_number}/{filename}` via `LocalStorageBackend`
- `StorageBackend.read(path) -> bytes` exists but is never exposed as API
- `MaterialVersionOut.storage_url` currently leaks local filesystem absolute paths to frontend
- Supported types: PDF, DOCX, XLSX (max 50MB)

## Tasks

### Task 1: Add download endpoint to materials router
**File:** `backend/app/api/materials.py`
- Add `GET /materials/{material_id}/versions/{version_id}/download` endpoint
- Accept query param `?mode=inline|attachment` (default: `attachment`)
- Look up `MaterialVersion` by ID, verify it belongs to the material
- Use `StorageBackend.read()` to get file bytes
- Return `Response` with:
  - `Content-Type` from `version.content_type`
  - `Content-Disposition: inline; filename="..."` for preview mode (PDF)
  - `Content-Disposition: attachment; filename="..."` for download mode
- Route MUST be placed before `/{material_id}` parameterized route (FastAPI gotcha #3)
- Require authentication (admin or user role)

### Task 2: Fix storage_url information disclosure
**File:** `backend/app/schemas/material.py`
- Add computed field `download_url` to `MaterialVersionOut` that generates `/api/v1/materials/{material_id}/versions/{id}/download`
- Either remove `storage_url` from the schema or make it `exclude=True` so it's not serialized to JSON
- Keep `storage_url` in the ORM model (needed for backend storage operations)

### Task 3: Add download endpoint tests
**File:** `backend/tests/test_materials_download.py` (new)
- Test download returns correct file bytes and Content-Type
- Test inline mode sets `Content-Disposition: inline`
- Test attachment mode sets `Content-Disposition: attachment`
- Test 404 for nonexistent material/version
- Test auth required (401 without token)
- Test storage_url is NOT in API response JSON
- Test download_url IS in API response JSON with correct format

```bash
cd backend && ruff check . && ruff format --check .
```
