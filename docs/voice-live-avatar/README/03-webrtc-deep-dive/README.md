## 3.1 WebRTC 连接的两个阶段（信令 + 媒体）

WebRTC 的连接建立并非一步完成，而是分为两个本质不同的阶段。理解这一点是掌握 WebRTC 架构的基础。

```
阶段1: 信令（Signaling）—— 需要"某种"通道
┌─────────┐                          ┌─────────┐
│ Client A │  ── SDP Offer ──────►   │ Client B │
│          │  ◄── SDP Answer ─────   │          │
│          │  ◄─► ICE Candidates ─►  │          │
└─────────┘                          └─────────┘
     ↕            通过什么传递？           ↕
   WebSocket / HTTP / MQTT / 甚至手动复制粘贴

阶段2: 媒体传输（Media）—— 纯 WebRTC，不需要 WebSocket
┌─────────┐  ◄══ SRTP/SCTP (音视频/数据) ══►  ┌─────────┐
│ Client A │         直接 P2P 或经 TURN         │ Client B │
└─────────┘                                     └─────────┘
```

**第一阶段——信令阶段**负责交换双方的媒体能力描述（SDP）和网络路径候选（ICE Candidates）。这个阶段需要一条预先存在的通信通道，但 WebRTC 标准**故意没有规定**这条通道使用什么协议。WebSocket 只是最常见的选择，HTTP POST、Server-Sent Events、甚至 Firebase 实时数据库都可以完成信令传递。

**第二阶段——媒体传输阶段**在信令完成后启动。此时音视频数据走的是 WebRTC 自己的 UDP/SRTP 通道，**完全不经过 WebSocket**。换言之，WebSocket 在这个阶段可以断开，媒体流仍然正常运行。

---

## 3.2 WebRTC 能否脱离 WebSocket 独立使用

许多初学者会产生一个疑问：WebRTC 是否能完全脱离 WebSocket？

答案是：**WebRTC 不能完全脱离信令通道，但信令不一定是 WebSocket。**

如上一节所述，WebRTC 的信令阶段必须依赖某种外部通道来交换 SDP 和 ICE 信息。但一旦信令完成、连接建立，媒体流就完全独立运行，不再需要信令通道参与。

一个形象的比喻：WebSocket 是 WebRTC 的"助产士"，不是"母体"——它帮助 WebRTC 连接诞生，但连接建立后，媒体流拥有完全独立的生命。

在 Voice Live 项目的实际架构中，WebSocket 不仅承担信令角色，还额外传输文本消息（语音转写文字、AI 回复文字），因此在整个会话生命周期中 WebSocket 始终保持连接。但从原理上讲，如果只需要音视频传输，WebSocket 在信令完成后即可关闭。

## 3.3 认证机制详解

WebRTC 协议本身没有"登录"或"身份认证"的概念。认证发生在**信令层**，由应用自行实现。整个认证流程分三步：

```
┌──────────────────────────────────────────────────┐
│                  认证流程                         │
├──────────────────────────────────────────────────┤
│                                                  │
│  1. 客户端 → 信令服务器: 带 API Key/JWT Token    │
│     （这一步是 HTTP 或 WebSocket，常规认证）       │
│                                                  │
│  2. 信令服务器验证身份后，返回：                    │
│     - SDP 信息（对方的媒体描述）                   │
│     - ICE 候选（网络路径信息）                     │
│     - TURN 凭据（临时用户名+密码）                 │
│                                                  │
│  3. WebRTC 连接建立时：                           │
│     - DTLS 握手（端到端加密，防中间人）             │
│     - TURN 凭据认证（如果用中继）                  │
│                                                  │
└──────────────────────────────────────────────────┘
```

**第一步**是常规的应用层认证。客户端通过 HTTP 或 WebSocket 向信令服务器发送凭据（API Key、JWT Token 等），这与普通 Web 应用的认证方式完全一致。

**第二步**由信令服务器在验证身份后，下发 WebRTC 建连所需的全部信息，包括 SDP、ICE 候选，以及供 TURN 服务器使用的临时凭据。

**第三步**发生在 WebRTC 连接建立过程中。DTLS 握手确保媒体流端到端加密，防止中间人攻击；如果使用了 TURN 中继，TURN 服务器还会验证第二步中下发的临时凭据。

在 Voice Live 中的具体实现是：认证通过 Azure API Key 在后端 WebSocket 连接时完成。前端不直接持有 Azure 凭据，而是由后端代理进行认证。

## 3.4 UDP 通信详解：双向还是单向？

这是一个经常被问到的问题：WebRTC 使用的 UDP 通道，到底是双向还是单向的？

**答案需要从两个层次来理解：传输层的 UDP 永远是双向的，但应用层的媒体流方向由 Transceiver direction 控制。**

```
┌─────────────────────────────────────────────────────────────┐
│ 传输层 (UDP)：永远是双向的                                    │
│                                                             │
│   即使你设了 recvonly，底层 UDP socket 仍然能收能发            │
│   因为 WebRTC 需要双向发送 DTLS 握手包、RTCP 控制包、         │
│   ICE keep-alive 心跳包等                                    │
│                                                             │
│   浏览器 ◄════ UDP 双向通道 ════► Azure Avatar               │
│          →  DTLS 握手包、RTCP 反馈、ICE 心跳                 │
│          ←  SRTP 视频包、SRTP 音频包、RTCP 统计              │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│ 应用层 (RTP 媒体)：由 Transceiver direction 控制             │
│                                                             │
│   "recvonly"  → 只接收对方的媒体流，不发送自己的媒体          │
│   "sendonly"  → 只发送自己的媒体流，不接收对方的              │
│   "sendrecv"  → 双向收发媒体流                               │
│                                                             │
│   本项目：direction = "recvonly"                              │
│   → 媒体数据单向（只收 Avatar 视频音频）                      │
│   → 但 UDP 通道本身仍然是双向的（控制包还是要发的）           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

在 Voice Live 项目中，浏览器端设置为 `recvonly`，即只接收 Azure Avatar 服务推送的视频和音频流。但这并不意味着 UDP 通道是单向的。以下是 `recvonly` 模式下 UDP 通道上实际传输的数据：

```
浏览器                                          Azure Avatar
  │                                                │
  │ ════════ 一条 UDP 双向通道 ════════════════════ │
  │                                                │
  │ ──► ICE Binding Request (心跳)                  │  ← 浏览器发出
  │ ◄── ICE Binding Response                       │  ← Azure 回
  │                                                │
  │ ──► RTCP Receiver Report (我收到了多少包)       │  ← 浏览器发出
  │ ◄── RTCP Sender Report (我发了多少包)           │  ← Azure 发出
  │                                                │
  │ ◄── SRTP Video (H.264 数字人视频帧)             │  ← 只有这个方向
  │ ◄── SRTP Audio (Opus 数字人音频帧)              │  ← 只有这个方向
  │                                                │
  │ ──► RTCP NACK (我丢了第 X 个包，请重传)         │  ← 浏览器发出
  │ ◄── SRTP 重传包                                │  ← Azure 补发
  │                                                │

总结：
  浏览器 → Azure: 控制包（心跳、反馈、重传请求）≈ 很少的数据
  Azure → 浏览器: 媒体包（视频+音频）+ 控制包 ≈ 大量数据

  "recvonly" 的含义是：媒体流单向，但 UDP 通道双向
```

简而言之：UDP 始终双向，`recvonly` 只是告诉 WebRTC "我不发媒体数据"，但控制信息（心跳、丢包反馈、DTLS 握手）仍然需要双向发送。这些控制信息虽然数据量很小，但对于维持连接健康、保障传输质量不可或缺。

## 3.5 TURN 中继时的协议封装

当两端都处于防火墙之后、无法直接建立 P2P 连接时，TURN 服务器作为中继介入。理解 TURN 的工作方式，首先要理解"为什么两端都是出站连接"这一关键设计。

### 为什么"都是出站"

防火墙的默认规则是"允许出站，阻止入站"。在没有 TURN 的情况下，两端互相连接意味着至少有一方需要接受入站连接，而这往往被防火墙拦截：

```
没有 TURN（直连尝试）：

  A 必须接受 B 的入站连接（或反过来）

  A ──入站──► B 的防火墙  ✗ 被拦截
  B ──入站──► A 的防火墙  ✗ 被拦截

  双方都在防火墙后面 → 谁也连不上谁


有 TURN（中继模式）：

  A 和 B 都不互相连接！
  双方都去连 TURN 服务器（出站连接）

  A ──出站──► TURN 服务器   ✓ A 的防火墙放行（因为是出站）
  B ──出站──► TURN 服务器   ✓ B 的防火墙放行（因为是出站）

  TURN 服务器在中间做转发：
    A 的数据 → TURN → 转发给 B
    B 的数据 → TURN → 转发给 A
```

TURN 把"两个入站问题"巧妙地转化为"两个出站连接 + 一个中间转发"，从而绕过双方的防火墙限制。

### TURN 的三种传输模式

TURN 并非只有一种工作模式。它提供三种传输方式，按优先级逐级降级：

```
┌────────────────────────────────────────────────────────────────────┐
│                    TURN 三种传输模式                                │
├─────────────┬──────────┬──────────┬────────────────────────────────┤
│ 模式         │ 端口     │ 协议     │ 说明                           │
├─────────────┼──────────┼──────────┼────────────────────────────────┤
│ TURN/UDP    │ 3478     │ UDP      │ 首选！客户端和TURN之间也走UDP   │
│ (最优)      │          │          │ 保留UDP全部性能优势             │
├─────────────┼──────────┼──────────┼────────────────────────────────┤
│ TURN/TCP    │ 3478     │ TCP      │ UDP被封时降级                   │
│ (降级)      │          │          │ 客户端→TURN走TCP，有一定损失    │
├─────────────┼──────────┼──────────┼────────────────────────────────┤
│ TURN/TLS    │ 443      │ TLS+TCP  │ 最后手段，伪装成HTTPS          │
│ (兜底)      │          │          │ 性能最差，但穿透力最强          │
└─────────────┴──────────┴──────────┴────────────────────────────────┘

ICE 会自动探测哪种能用，优先选最快的。
```

三种模式的实际数据路径如下：

```
模式 1: TURN/UDP（最优情况）
┌──────┐  UDP  ┌──────────┐  UDP  ┌──────┐
│ 浏览器 │◄════►│ TURN 服务器│◄════►│ Azure│
└──────┘       └──────────┘       └──────┘
  全程 UDP，性能几乎等同于直连（只多了一跳延迟）


模式 2: TURN/TCP（UDP 被封时）
┌──────┐  TCP  ┌──────────┐  UDP  ┌──────┐
│ 浏览器 │◄────►│ TURN 服务器│◄════►│ Azure│
└──────┘       └──────────┘       └──────┘
  浏览器→TURN 走 TCP（穿防火墙）
  TURN→Azure 仍走 UDP（中继内部转发）
  ⚠️ 浏览器侧有 TCP 开销


模式 3: TURN/TLS (port 443)（最后手段）
┌──────┐ TLS  ┌──────────┐  UDP  ┌──────┐
│ 浏览器 │◄────►│ TURN 服务器│◄════►│ Azure│
└──────┘ :443  └──────────┘       └──────┘
  浏览器→TURN 走 TLS+TCP（看起来像 HTTPS）
  TURN→Azure 仍走 UDP
  ⚠️ 浏览器侧有 TCP + TLS 双重开销
```

这里有一个非常重要的细节：**TURN 服务器是一个协议转换节点，而不是简单的隧道。** 它的工作流程是：

1. **终结**客户端的连接（TCP 或 UDP）
2. **解包**数据
3. 用**新的 UDP 连接**转发给对端

所以即使客户端到 TURN 走了 TCP，TURN 到对端仍然可以走 UDP。这意味着 TCP 带来的性能损失只存在于客户端到 TURN 这一段，而非全程。

## 3.6 TCP 封装后 UDP 的性能影响

当 TURN 使用 TCP 或 TLS 模式时，UDP 原本的实时性能优势会受到多大影响？这需要逐项分析 UDP 对实时音视频的三个核心优势，以及 TCP 封装后各自的损失程度。

```
┌──────────────────────────────────────────────────────────────────┐
│              UDP 的三大实时优势 vs TCP 封装后的影响               │
├────────────────┬─────────────────┬───────────────────────────────┤
│ UDP 优势        │ TCP 封装后       │ 影响程度                      │
├────────────────┼─────────────────┼───────────────────────────────┤
│ 1. 无队头阻塞   │ ❌ 丧失          │ 严重                          │
│   (丢一个包不   │ TCP 丢包会阻塞   │ 网络波动时可能出现             │
│    影响后续包)  │ 后续所有数据     │ 突然卡顿→快速恢复的现象       │
├────────────────┼─────────────────┼───────────────────────────────┤
│ 2. 低延迟       │ ⚠️ 部分丧失      │ 中等                          │
│   (无重传等待)  │ TCP 重传增加     │ 增加 ~50-150ms 尾部延迟       │
│                │ ~50-150ms       │ 平均延迟增加不大               │
├────────────────┼─────────────────┼───────────────────────────────┤
│ 3. 可丢包       │ ❌ 丧失          │ 中等                          │
│   (过时的帧可以 │ TCP 保证必达     │ 但 WebRTC 的 jitter buffer   │
│    直接丢弃)    │ 即使已过时也重传 │ 可以在应用层补偿               │
└────────────────┴─────────────────┴───────────────────────────────┘
```

其中**队头阻塞（Head-of-Line Blocking）**是最严重的问题。通过以下对比可以直观感受差异：

```
UDP 直连（理想情况）：
  帧1 ✓ → 播放
  帧2 ✗ → 丢了，跳过！直接播帧3     ← UDP 允许跳过
  帧3 ✓ → 播放
  帧4 ✓ → 播放
  结果：帧2丢了，画面闪一下，但后续帧不受影响

TCP 封装（TURN/TCP 或 TURN/TLS）：
  帧1 ✓ → 播放
  帧2 ✗ → 丢了... TCP 自动重传...
  帧3   → 等着... 帧2还没到...      ← TCP 队头阻塞！
  帧4   → 继续等...
  帧2 ✓ → 终于到了（但已经过时了）
  帧3 ✓ → 可以播了
  帧4 ✓ → 可以播了
  结果：帧2丢了，帧3和帧4全被阻塞，卡顿 ~100-300ms
```

在 UDP 直连模式下，丢失一帧只导致画面短暂闪烁，后续帧不受任何影响。而在 TCP 封装下，一帧丢失会导致后续所有帧排队等待重传完成，造成可感知的卡顿。这就是为什么 TURN 优先选择 UDP 模式，只在 UDP 完全不可用时才降级到 TCP。

## 3.7 各种连接模式的性能对比

下表汇总了所有连接模式的实际性能表现，帮助读者建立全局认知：

```
┌──────────────┬────────────┬────────────┬─────────────────────┐
│ 连接模式      │ 额外延迟    │ 丢包表现    │ 实际体验            │
├──────────────┼────────────┼────────────┼─────────────────────┤
│ 直连 UDP     │ 基准 0ms   │ 最优       │ 最流畅              │
│ (无NAT问题)  │            │ 丢帧跳过   │                     │
├──────────────┼────────────┼────────────┼─────────────────────┤
│ TURN/UDP     │ +10-30ms   │ 最优       │ 几乎无感知差异       │
│ (只多一跳)   │ (一跳延迟) │ 丢帧跳过   │ 和直连体验一样       │
├──────────────┼────────────┼────────────┼─────────────────────┤
│ TURN/TCP     │ +30-80ms   │ 一般       │ 正常情况下OK         │
│              │            │ 队头阻塞   │ 弱网下偶尔卡顿       │
├──────────────┼────────────┼────────────┼─────────────────────┤
│ TURN/TLS:443 │ +50-150ms  │ 较差       │ 能用但有感知延迟     │
│ (最后手段)   │ (TLS握手+  │ 队头阻塞+  │ 弱网下卡顿更明显     │
│              │  TCP开销)  │ 加密开销   │                     │
├──────────────┼────────────┼────────────┼─────────────────────┤
│ 无连接       │ ∞          │ N/A        │ ❌ 黑屏，什么都没有  │
└──────────────┴────────────┴────────────┴─────────────────────┘

结论：TURN/TLS:443 (最差情况) 的体验 >>>>> 无连接
```

这张表传递了一个重要信息：即使是性能最差的 TURN/TLS 模式，其体验也远远优于完全没有连接。TURN 的降级策略本质上是"有总比没有好"——在网络条件允许时追求最优性能，在条件受限时保障基本可用性。

## 3.8 WebRTC 的补偿机制

WebRTC 并非只依赖底层传输协议来保障体验。即使底层走了 TCP，WebRTC 在应用层仍然内置了多重补偿机制来缓解性能损失：

```
┌──────────────────────────────────────────────────────────────┐
│ WebRTC 自带的补偿机制                                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ 1. Jitter Buffer（抖动缓冲）                                 │
│    → 不是收到就立即播，而是缓冲 50-200ms                      │
│    → 用缓冲平滑网络波动造成的帧到达时间不均匀                 │
│    → TCP 队头阻塞导致的延迟波动被缓冲吸收                     │
│                                                              │
│ 2. 自适应码率（Adaptive Bitrate）                            │
│    → 检测到网络差 → 自动降低视频分辨率/帧率                   │
│    → 减少数据量 → 减轻 TCP 拥塞                              │
│                                                              │
│ 3. FEC（前向纠错）                                           │
│    → 发送冗余数据包，接收端可以从部分数据恢复完整帧           │
│    → 减少重传需求 → 降低 TCP 队头阻塞的触发频率               │
│                                                              │
│ 4. 关键帧请求                                                │
│    → 丢了太多帧 → 请求发送端发一个完整关键帧 (I-frame)        │
│    → 快速恢复画面，而不是等所有丢失帧重传完                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**Jitter Buffer** 是最基础的补偿手段。它在接收端维护一个 50-200ms 的缓冲区，不是收到数据包就立即播放，而是先缓冲再按时间戳顺序输出。这样一来，TCP 队头阻塞导致的短暂延迟波动可以被缓冲区吸收，用户感知到的是平滑连续的播放。

**自适应码率**通过实时检测网络状况，在网络变差时自动降低视频分辨率和帧率，减少数据量以缓解拥塞。

**FEC（前向纠错）**在发送端额外附加冗余数据包，即使部分包丢失，接收端也能从剩余数据中恢复完整帧，从而减少重传需求。

**关键帧请求**是最后的恢复手段。当连续丢失太多帧导致画面无法正常解码时，接收端会主动请求发送端发送一个完整的 I-frame（关键帧），从而快速恢复画面，而不是被动等待所有丢失帧逐一重传。

## 本章小结

本章从 WebRTC 的连接生命周期出发，深入探讨了以下核心要点：

- WebRTC 连接分为**信令**和**媒体**两个独立阶段，信令通道不限于 WebSocket
- 认证发生在**信令层**，WebRTC 本身不处理身份验证
- UDP 通道在传输层**始终双向**，`recvonly` 只控制应用层媒体流方向
- TURN 服务器是**协议转换节点**，提供 UDP/TCP/TLS 三种模式逐级降级
- TCP 封装主要损失在于**队头阻塞**，但 WebRTC 的 Jitter Buffer、自适应码率、FEC 和关键帧请求等机制提供了有效补偿
- 即使是最差的 TURN/TLS 模式，体验也远远优于完全没有连接

理解这些底层机制，将帮助你在下一章的 NAT 穿透学习中更好地把握 ICE、STUN、TURN 三者的协作关系。
